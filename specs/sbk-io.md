# .sbk 定跡フォーマット入出力仕様

## 基本方針

.sbk を単純に連想配列へ展開すると 100MB 程度のファイルでも 1 GB 以上のメモリを消費してしまう。
Electron のバックグラウンドプロセスで 2GB くらいまでしか安全にメモリを確保できないため、 200MB 程度のファイルを扱うことは困難である。
そこで、メモリ上には Packed-SFEN とファイル内のオフセットの情報を ArrayBuffer で持つことにより、規模の大きい定跡ファイルを安全に利用可能にする。

## インメモリデータ設計

インメモリには以下のデータを持つ。

- 生の .sbk ファイル (Protocol Buffers) のバイト列: ArrayBuffer
- 生の .sbk のオフセットを管理するテーブル: ArrayBuffer
- 編集したデータの連想配列: Map

## 定跡の検索

定跡を検索する時は編集したデータの連想配列を先に参照し、データがない場合のみ Protocol Buffers のデータを参照する。
検索順位の考え方は、他の定跡フォーマットの On-the-fly モードと同様である。

## 読み込み時のテーブル構築処理

読み込みの初期段階では .sbk の要素を先頭から順に参照し、同じ順序でメモリ上に以下のデータを配置する。

1. 256bits : Packed-SFEN
2. 39bits : .sbk ファイル内の Offset
3. 1bit : 書き出し時のみ使用するルートノード判定フラグ

Offset は `SBookState` の field tag (`BookStates` の 1 要素先頭) の開始位置を指す。

ただし、 Packed-SFEN はこの時点では確定できないため 0 値で初期化する。
.sbk のオフセットは Protocol Buffers の Tag の位置を指すものとする。
ルートノード判定フラグは書き出し時まで使用しない。

全てのデータを 1 つの ArrayBuffer に並べることにより、メモリ消費を最適化する。

.sbk に SFEN が書かれている局面から探索をして各局面の Packed-SFEN を埋めていく。

最後に Packed-SFEN でソートし、二分探索が可能な状態を作る。

## 書き出し時の処理

ノードの繋がりを再探索して SFEN を書き出すべきルートノードを特定する。
最初にルートノード判定フラグを全て 1 にして、ルートノードではないことが確定したものを 0 にしていく。
ノードに循環がある場合にルートノードが 1 つも残らない状況を作らないようにする。
循環がある場合には速度を優先し、正しいルートノードの数よりも多くのノードがルートとして判定されることを許容する。

## 保存制約

On-the-fly で開いている .sbk は同一パス上書きを禁止し、別名保存のみ許可する。

Packed-SFEN がゼロ値のままの要素は無視する。
ソートによりゼロ値の要素は先頭に集まるため、その範囲を検出して全ての処理でスキップする。
